## 1. Introduction — Problem Recap and Approach Summary

Breadth-First Search (BFS) explores a graph level by level and underpins social-network analysis,
web crawling, and route planning. Sequential BFS is inherently serial — nodes are expanded one at a
time — so on graphs with millions of nodes and 10⁸ edges (e.g. SNAP *com-Orkut*: 3,072,441 nodes,
117,185,083 edges) traversal becomes a bottleneck.

The parallelization opportunity is **frontier-level parallelism**: all nodes at the same BFS level
are mutually independent and can be expanded concurrently. We implement two engines sharing one
immutable CSR graph:

- a **sequential baseline** (primitive `int[]` ring-buffer queue), and
- a **parallel BFS** using `java.util.concurrent` — an `ExecutorService` thread pool, `Callable`
  tasks per frontier partition, a `Future.get()` barrier between levels, and a lock-free
  `AtomicIntegerArray` visited set claimed via `compareAndSet`.

On top of the algorithms we built a **live demonstration platform** (Javalin WebSocket/REST backend
+ React dashboard) so the effect of parallelism is visible in real time.

## 2. Sequential Baseline Implementation

`SequentialBFS` (`src/main/java/edu/gazi/ceng479/bfs/bfs/SequentialBFS.java`) initialises a `level[]`
array to −1 (which also serves as the visited marker), enqueues the source, then repeatedly dequeues
a node and enqueues its unvisited neighbours. To keep the baseline honest (a slow baseline would
inflate speedup), the queue is a **primitive `int[]` ring buffer** with head/tail indices — no
`Queue<Integer>` autoboxing. Complexity: **O(V + E)** time, O(V) space; each edge is examined once.

The graph is stored in **Compressed Sparse Row (CSR)** form — two `int[]` arrays (`rowPtr`, `colIdx`)
— giving contiguous, cache-friendly neighbour iteration and avoiding the >3.7 GB an
`List<List<Integer>>` would need on Orkut.

## 3. Parallel Implementation — Thread Design

`ParallelBFS` restructures BFS around the frontier (`src/.../bfs/ParallelBFS.java`, Listing 2 of the
design):

1. The current frontier (an `int[]`) is split into contiguous, near-equal ranges by
   `FrontierPartitioner`, one per worker thread.
2. Each range is submitted as a `Callable<IntArrayList>` to a **fixed thread pool created once per
   run**; each worker scans its nodes' neighbours and **claims** unvisited ones with
   `visited.compareAndSet(nbr, 0, 1)`, appending winners to a *thread-local* next-frontier list.
3. A **`Future.get()` barrier** collects the per-thread lists into the next frontier before the next
   level begins.

**Correctness (no races).** `compareAndSet` guarantees each node is claimed exactly once; the
`level[]`/`parent[]` writes are disjoint (only the CAS winner writes a given index), satisfying
Bernstein's conditions; the barrier preserves level ordering. The parallel BFS therefore produces an
**identical level array** to the sequential reference — enforced at runtime by `ResultVerifier` and
in tests by 50+ random-graph equivalence checks plus a 100-iteration high-contention stress test.

**Two optimizations (documented as ADRs in `context.md`):**
- *Test-and-test-and-set* — a cheap volatile `isVisited` read before the expensive CAS. On dense
  graphs most claims fail (neighbour already visited); the read keeps the cache line shared instead
  of bouncing it between cores. This lifted dense 4-thread speedup from 1.12× to 2.20×.
- *Small-frontier serial fallback* — frontiers below a cutoff are expanded on the calling thread,
  avoiding pool overhead on the many tiny levels of sparse graphs.

## 4. Performance Comparison — Speedup Tables and Graphs

Hardware: Intel-class CPU, **4 physical / 8 logical cores**, 16 GB RAM, JDK 21 (release-17
bytecode), Windows. Each value is the mean of **5 timed repetitions** after **2 warm-up**
repetitions (to absorb JIT/GC), timing the BFS region only.

**Table 2 — Framework-overhead baseline.** Parallel-with-1-thread is slower than pure sequential
(dense 140.6 ms vs 114.6 ms; 0.81×) because of the atomic visited array and per-level merge — the
expected cost the parallelism must overcome. Reported to isolate framework overhead from genuine
parallel gains (proposal §6).

**Figures** (`docs/figures/`, from `scripts/plot.py`):

**Discussion.** Speedup is monotonic and **dense graphs scale better than sparse** (3.18× vs 2.67×
at 8 threads), because dense graphs amortize the per-level coordination (merge + barrier) over much
larger frontiers, shrinking the sequential fraction — exactly the Amdahl intuition (p=0.90, sequential
fraction ≈ 0.10). Measured speedups stay below the Amdahl ceiling, consistent with (a) only 4
physical cores (8 logical via Hyper-Threading), so scaling past 4 threads yields diminishing returns,
and (b) load imbalance from skewed degree distributions and atomic-array coherence traffic. No
super-linear effects were observed.

## 5. Academic Background — Brief Review of Related Work

Frontier-level parallel BFS is well studied. Bader and Madduri (2006) designed multithreaded BFS for
the Cray MTA-2. Hong et al. (2011) accelerated graph algorithms by mapping frontier expansion to
warps/threads and highlighted the irregular-memory challenge that makes shared-memory CPU threading a
good fit for pointer-chasing traversal. Beamer et al. (2012) introduced direction-optimizing
(top-down/bottom-up hybrid) BFS; we implement the top-down variant and list bottom-up as future work.
Amdahl's Law (1967) frames the expected speedup given a fixed sequential fraction. Benchmark graphs
come from the SNAP collection (Leskovec & Krevl, 2014).

## 6. Challenges and Solutions

| Challenge | Solution |
|-----------|----------|
| Race conditions on the shared visited state | Lock-free `AtomicIntegerArray.compareAndSet` claim; disjoint level/parent writes; verified by equivalence + 100-iteration stress tests. |
| Poor dense-graph scaling (atomic contention) | Test-and-test-and-set: cheap read before CAS — dense 4t 1.12×→2.26×. |
| Overhead dominating sparse small frontiers | Small-frontier serial fallback below a cutoff. |
| Honest measurement vs live visualization | Separate **clean** (`run`) and **observed** (`runObserved`) paths — instrumentation never touches benchmark timing. |
| Slow baseline would inflate speedup | Primitive `int[]` ring-buffer queue, no autoboxing. |
| Loading 117M-edge SNAP graphs | Two-pass CSR build with id remapping; `-Xmx`; `--max-edges` subgraph cap. |
| **Turkish-locale `toUpperCase()`** mapping `i`→`İ` broke `--mode ui`/`verify` | Use `Locale.ROOT` for CLI case folding; regression-tested. |

## 7. Conclusion and Future Improvements

Parallel BFS achieved up to **3.18× speedup** (dense, 8 threads) over a fast sequential baseline,
producing bit-identical traversal results, on a 4-physical-core machine — consistent with Amdahl's
Law and the literature. The accompanying live platform makes the speedup, workload distribution, and
sequential-vs-parallel difference visible in real time.

Future work: bottom-up / direction-optimizing BFS (Beamer et al.); edge-count-balanced frontier
partitioning to counter degree skew; NUMA-aware pinning; and a CUDA port for comparison.
